\noindent\textbf{Abstract:}
\noindent This thesis deals with the formalization of Polonius, the latest version of the borrow checker in the Rust programming language. Rust implements memory safety through an ownership system that prevents errors such as dangling pointers and use-after-free. While previous versions of the checker have been well documented, Polonius exists primarily in the form of blog posts and an experimental implementation. This thesis attempts to mathematically describe Polonius by defining basic sets, relations, and rules.

\chapter{Uvod}
Pomnilniška varnost \angl{memory safety} je na področju pisanja programske opreme vedno aktualna tema. Microsoft je pokazal, da so napake pri upravljanju s pomnilnikom najbolj pogost tip napak\cite{Microsoft70Percent}. Pri projektu Chromium, ki je osnova za Google Chrome, so opazili, da okoli 70 procentov hroščev povzročijo tovrstne napake \cite{MemorySafetya}. Očitno je, da če bi se znebili te kategorije napak, bi lahko rešili velik del hroščev. Posledično so se pomnilniške varnosti razvijalci jezikov lotili na različne načine.

Eden najbolj uporabljenih jezikov je C, kjer je programerju povsem prepuščeno upravljanje s pomnilnikom.
Tovrsten pristop, imenovan ročno upravljanje s pomnilnikom, lahko vodi do izredno hitrih programov in kratkih časov prevajanja (v primerjavi z bolj kompleksnimi jeziki, kot je Rust \cite{glazarCodingRustBad2023}), ampak je tudi vir številnih napak \cite{MemorySafetya}.

Alternativni pristop ročnemu upravljanju je avtomatsko upravljanje s pomnilnikom \angl{garbage collection}, kjer programski jezik zagotavlja varno dodeljevanje in sproščanje pomnilnika ter s tem razbremeni programerja, da se lahko osredotoči na pisanje programa. Vendar ta način ni brez kompromisa, saj imajo jeziki z avtomatskim upravljanjem s pomnilnikom večjo porabo pomnilnika zaradi zakasnjenega sproščanja. Poleg tega se pojavijo premori med izvajanjem programa ali pa zakasnitve pred vsako operacijo, da lahko čistilec pomnilnika najde pomnilniške lokacije za sprostitev \cite{bakerListProcessingReal1978}.

Rust se upravljanja s pomnilnikom loti na drugačen način. Veljavnost dostopanja do pomnilniških lokacij se preverja med časom prevajanja s pomočjo preverjevalnika izposoj \angl{borrow checker}. To je del Rustovega prevajalnika, ki se ukvarja s tokom podatkov in pomnilniškimi lokacijami. Rust imenuje zbirko pravil, ki opisuje delovanje preverjevalnika izposoj, lastništvo \angl{ownership}. Tak pristop nam teoretično, omogoči najboljše obeh svetov: zagotovilo, da je naš program pomnilniško varen, ki nam ga da avtomatično upravljanje s pomnilnikom ter hitrost izvajanja programov jezikov, ki ga lahko dosežemo z ročnim upravljanjem pomnilnika \cite{klabnikRustProgrammingLanguage2023}. Pogosto omenjena slabost Rusta je pa dolg čas prevajanja \cite{glazarCodingRustBad2023}, ki sicer ni samo odvisen od preverjevalnika izposoj, vendar zagotovo njegov prispevek ni zanemarljiv.

V nadaljevanju bomo smatrali, da pojma \textit{varen program} in \textit{pravilen program} pomenita tak program, ki ne povzroča napak prip upravljanju s pomnilnikom ne glede na to, ali je prevajalnik zaradi težav z izračunljivostjo to sposoben ugotoviti ali pa ne.

Preverjevalnik izposoj se je med razvojem Rusta bistveno spremenil od svoje prvotne implementacije. Na začetku je bil dokaj preprost in ni sprejel veliko pravilnih programov zaradi svoje konzervativnosti pri zagotavljanju varnosti \cite{2094nllRustRFC}. Zato se je čez par let pojavila naslednja različica preverjevalnika, imenovana NLL (\textit{non-lexical lifetimes}), ki je rešila veliko pogostih problemov s prvotno različico. Vendar NLL še vedno ne sprejema vseh veljavnih programov. Da bi rešili probleme NLL-ja, so Rustovi razvijalci predlagali najnovejšo različico preverjevalnika imenovano Polonius, ki drugače zastavi problem lastništva in tako sprejme še večji delež pravilnih programov \cite{AliasbasedFormulationBorrow}.

NLL je bil prvotno natančno opisan v RFC-ju z zelo točnim jezikom, kar je potem vodilo njegov razvoj. Polonius pa je prvotno nastal kot predlog na spletnem blogu in se počasi razvijal s pomočjo dodatnih objav na blogu enega izmed Rustovih razvijalcev \cite{PoloniusRevisitedParta,PoloniusRevisitedPart,WhatPoloniusPolonius}. Do danes ne obstaja celovit centraliziran formalen opis Poloniusa, le nekaj spletnih objav, delni formalni opis v magisterskem delu enega izmed razvijalcev \cite{stjernaModellingRustsReference}, nedokončana knjiga \cite{WhatPoloniusPolonius} ter trenutna implementacija v Rustovem prevajalniku.

Cilj te naloge je potemtakem na svoj način formalizirati inferenčna pravila, ki sestavljajo Polonius, in nadgraditi nekaj že vzpostavljenih formalizacij \cite{stjernaModellingRustsReference}. Najprej bomo raziskali pretekle poskuse formalizacije Rusta in sorodne načine upravljanja s pomnilnikom. Nato sledi intuitivni opis Rustovih pravili izposojanja ter bomo nato končali s formalizacijo Poloniusa ter pravil preverjevalnika izposoj.

\section{Motivacijski primer}

Za grajenje intuicije o razlikah med trenutno različico preverjevalnika izposoj (NLL) in med naslednjo (Polonius) bomo obravnavali motivacijski primer, ki ga NLL zavrne in Polonius sprejme. Primer je prilagojen iz prvotnega predloga NLL-ja \cite{2094nllRustRFC}.

\begin{listing}[ht!]
	\begin{minted}[linenos]{rust}
fn process(val: &mut String) {
    unimplemented!();
}

fn process_or_default<'a>(map: &'a mut HashMap<&str, String>) 
        -> &'a mut String {
    let key = "test";
    match map.get_mut(&key) { // ------------------+ 'lifetime
        Some(value) => value,                   // |
        None => {                               // |
            map.insert(key, String::default()); // |
            //  ^~~~~~ ERROR.                   // |
            map.get_mut(&key).unwrap()          // |
        }                                       // |
    } // <-----------------------------------------+
}
    \end{minted}
	\caption{Motivacijski primer za Polonius}
	\label{listing:mot_ex}
\end{listing}

\begin{listing}[ht!]
	\begin{minted}{text}
error[E0499]: cannot borrow `*map` as mutable more than once at a time
  --> /tmp/IWXsFebCZD/main.rs:11:13
   |
5  |   fn process_or_default<'a>(map: &'a mut HashMap<&str, String>) 
   |                         -- lifetime `'a` defined here
...
8  |       match map.get_mut(&key) { // ------------------+ 'lifetime
   |       -     --- first mutable borrow occurs here
   |  _____|
   | |
9  | |         Some(value) => value,                   // |
10 | |         None => {                               // |
11 | |             map.insert(key, String::default()); // |

...  |
14 | |         }                                       // |
15 | |     } // <-----------------------------------------+
   | |_____- returning this value requires that `*map` is borrowed for `'a`
    \end{minted}
	\caption{Napaka pri prevajanju primera \ref{listing:mot_ex}}
	\label{listing:mot_ex_err}
\end{listing}

Če postopoma sledimo sporočilu o napaki na izpisu \ref{listing:mot_ex_err}, lahko vidimo kje NLL ni zmožen sprejeti pravilnega programa. Na vrstici 8 kličemo funkcijo \verb|get_mut|, ki vrne unijo z dvem možnostima. Lahko vrne spremenljivo referenco na vrednost, ki pripada ključu, ali pa ne vrne nič (\verb|None|). Če vrne vrednost, se šteje, kot da je spremenljivka \verb|map| začasno izsposojena (torej obstaja spremenljiva referenca na njene podatke), kar se zgodi v vrstici 9. Vendar Rust-ov prevajalnik smatra, da je \verb|map| še vedno izsposojena, tudi če ne vrnemo reference iz funkcije \verb|get_mut| (vrstice 11-13). Ko torej poskušamo vstaviti nov par, ki je sestavljen iz vrednosti in ključa, nam to Rustov prevajalnik konzervativno prepreči, saj operacija \verb|insert| zahteva spremenljivo referenco, dve spremenljivi referenci na isto mesto pa po pravilih preverjevalnika izposoj ne smeta obstajati.

Pri primeru \ref{listing:mot_ex}, kjer se NLL ne sprejme pravilnega programa, pa ga Polonius pravilno prevede. Namreč ima večje zmožnosti sledenja kontrolnemu toku in lahko zgornjo analizo opravi bolje. NLL ima trenutno omejene zmožnosti obravnavanja kontrolnega toka, ki jih Polonius nadgradi v zameno za hitrost. Amanda Stjerna, ena izmed razvijalcev Poloniusa, je na predstavitvi na konferenci EuroRust omenila, da je plan v prihodnosti sestaviti dvoslojni preverjevalnik izposoj. \section{Razvoj Rustovega preverjevalnika izposoj}
Rustov v primerjavi z sorodnimi sistemskimi jeziki kot sta C in C++ implementira kot del procesa prevajanja
preverjevalnik izposoj. Ta skrbi, da se nikoli ne uporabi neveljavna pomnilniška lokacija. S tem namenom
enforca tri pravila:
- Vsak vrednost v Rustu ima lastnika.
- Vsaka vrednost ima lahko natanko enega lastnika hkrati.
- Ko lastnik goes out of scope, se vrednost sprosti.

Zaradi zgornjih pravil pravimo temu modelu \textbf{lastništvo} (\textit{ownership}) \cite{klabnikRustProgrammingLanguage2023}.
Lastnik je spremenljivka oziroma binding na katerega je ta vrednost vezana. V spodnjem preprostem
primeru opazimo, da je vrednost "hello" enkrat zamenja lastnika, torej njen prvotni lastnik a ni
potem več veljaven. Če hočemo uporabiti a po tem, ko ni več lastnik vrednosti, nam prevajalnik vrne napako.
% preveri ali je to res

\begin{listing}[!ht]
	\begin{minted}{rust}
let a = "hello";
let b = a;
println("{}", a); // vrne napako
\end{minted}
	\caption{Primer lastništva}
	\label{listing:ownership1}
\end{listing}

Koncept scopa pa lahko preprosto prikažemo z leksičnim scopom, tako da inicializiramo novo vrednost a-ja
znotraj gnezdenega bloka, ki ustvari nov scope.

\begin{listing}[!ht]

V zgornjih primerih nismo videli večje razlike med Rustom in sorodnimi jeziki. Vendar kjer se pa Rust razlikuje
je pa v definicijah referenc in razdelitvi teh na tipe. Ko v Rustu govorimo o referencah, lahko rečemo, da so na
prvi pogled podobne kazalcem, kakršne poznamo iz drugih programskih jezikov. A ključna razlika je v tem, da prevajalnik
v Rustu poskrbi, da takšna referenca vedno kaže na veljavno vrednost pravega tipa – in to skozi celotno življenjsko
dobo te reference \cite{klabnikRustProgrammingLanguage2023}. Ta varnostni mehanizem nam omogoča nekaj, kar je v
mnogih drugih jezikih bistveno težje doseči: gotovost, da reference "ne visijo v prazno" in da ne dostopamo do
podatkov, ki morda sploh več ne obstajajo.

Te reference pa niso vse enake. Ločimo jih v dve glavni skupini \cite{crichtonGroundedConceptualModel2023,yanovskiGhostCellSeparatingPermissions2021,weissOxideEssenceRust2019}.

a ta nam daje možnost, da z vrednostjo, na katero kaže, tudi poljubno manipuliramo. Te refe

Pravilo \verb|Use-Init| nam pove, da lahko uporabljamo samo spremenljivke, ki so zagotovo inicializirane
na točki v programu, kjer jih uporabljamo. Glede na prvotno implementacije Poloniusa Rustove ekipe poleg
osnovnih dejstev opisanih v spletni objavi dobimo še dejstva \verb|var_defined_at|, \verb|var_used_at| in
\verb|var_dropped_at| \cite{RustlangPoloniusDefines}.

Mogoče tole zgoraj ni tako relevantno, ker se ukvarjamo z implementacijo že. Lahko definiramo množico
$Poti(p)$, ki nam poda vse poti skozi graf poteka od začetka funkcije do začetka trenutne točke $p$
v programu. Potem lahko definiramo še predikat $Init(\pi, x, p)$, ki velja natanko tedaj, ko je spremenljivka
$x$ skozi pot $\pi$ definirana na točki $p$. Končno pravilo se nato glasi:
\begin{equation}
	\text{Use-Init}(x, p) \iff \forall \pi \in \text{Poti}(p): \text{Init}(\pi, x, p)
\end{equation}

% https://rustc-dev-guide.rust-lang.org/borrow_check/moves_and_initialization/move_paths.html za definicijo move path
Opomba o mestih in poteh premika (\textit{move path}): pojem predpone (\textit{prefix}) je načeloma definiran za pot
premika, vendar ga tukaj posplošimo na mesta iz MIR, ker se nam bolj sklada s terminologijo, ki jo uporabljamo.
Rustov priročnik za prevajalnik tudi omeni, da sta ta pojma približno enaka. Uporabljamo ga namesto spremenljivk
zaradi tega, ker nam opiše lahko bolj fino dostopne podatke kot so polja struktov.

Za pravilo Move-Deinit moramo vpeljati še dva predikata. Prvi nam bo povedal ali se mesti
prekrivata t.j. ali je katero mesto predpona drugega (v Rustovem prevajalniku se temu reče \textit{prefix}). Poimenovali
ga bomo $Prekrivanje(m_{1}, m_{2})$, ki velja natanko tedaj, ko je $m_{1}$ predpona $m_{2}$ ali obratno. Torej
$Prekrivanje(tuple.0, tuple.0.1)$ bi veljalo, $Prekrivanje(tuple.0, tuple.1)$ pa ne.

Naslednji predikat, ki ga uvedemo je $Premaknjen(\pi, m, p)$, ki velja natanko tedaj, ko je bilo mesto $m$ premaknjeno
pred točko $p$ na poti $\pi$. Kaj točno je premik moram še malo pobrskati po dokumentaciji ampak konceptualno to pomeni,
da lastnik ni več $m$ ampak nekdo drug. Torej pravilo velja ko:

\begin{equation}
	\begin{aligned}
		 & \text{Move-Deinit}(m, p) \iff \\ &\nexists \pi \in \text{Poti}(p), m_{2}: \text{Prekrivanje}(m, m_{2}) \land \text{Premaknjen}(\pi, m_{2}, p)
	\end{aligned}
\end{equation}

Za naslednji pravili Shared-Readonly in Unique-Write moramo definirati še nekaj predikatov. Ker se tokrat
ukvarjamo s posojami, moramo razločevati met deljenimi in spremenljivimi (unikatnimi). V ta namen vpeljemo
predikat $\text{VrstaPosoje}(L) \in \{shrd, uniq\}$. Da pa lahko razločimo med aktivnimi in preteklimi posojami
pa ustvarimo predikat $\text{PosojaAktivna}(L,p)$, ki velja natanko tedaj, ko je posoja $L$ aktivna na točki $p$.
Mesto, ki je bilo sposojeno z posojo $L$ označimo z $O(L)$.

Poleg predikatov za posoje pa nam še manjkajo predikati, ki opisujejo operacije, ki se izvajajo nad mesti.
Intuitivno je to lahko več različnih operacij vendar nas zanimajo dve vrsti. Take, ki bi razveljavile deljeno
posojo označimo z $\text{RazveljaviDeljeno}(m,p)$ in velja natanko tedaj, ko se v točki $p$ nad mestom $m$ izvede
taka operacija, ki bi lahko razveljavila posojo, ki si sposoja iz mesta $m$ (to bi bilo ali pisanje v mesto $m$ ali
pa ustvarjanje spremenljive posoje).

Na podoben način definiramo $\text{RazveljaviSpremenljivo}(m,p)$, ki velja ko je operacija taka, ki razveljavi
spremenljivo posojo (ustvarjanje kakršnekoli nove posoje, pisanje v mesto, branje iz mesta).
Tako lahko sestavimo naši naslednji dve pravili:
\begin{equation}
	\begin{aligned}
		\text{Shared-Readonly}(p)   & \iff  \nexists L,m:
		\\ \text{PosojaAktivna}(L,p) &\land \text{VrstaPosoje} = shrd \land \\
		\text{Prekrivanje}(m, O(L)) & \land \text{RazveljaviDeljeno}(m,p)
	\end{aligned}
\end{equation}
\begin{equation}
	\begin{aligned}
		\text{Unique-Write}(p)      & \iff  \nexists L,m:
		\\ \text{PosojaAktivna}(L,p) &\land \text{VrstaPosoje} = uniq \land \\
		\text{Prekrivanje}(m, O(L)) & \land \text{RazveljaviSpremenljivo}(m,p)
	\end{aligned}
\end{equation}

Za zadnje pravilo potrebujemo še en predikat imenovan $\text{MestoAktivno}(m,p)$, ki velja natanko tedaj,
ko je mesto $m$ še aktivno (torej ni bilo dropped) na točki $p$. Potem pravilo Ref-Live lahko zapišemo tako:
\begin{equation}
	\begin{aligned}
		\text{Ref-Live}(p) & \iff \nexists L, m: \\ \text{PosojaAktivna}(L,p) &\land \text{Prekrivanje}(m, O(L)) \land
		\neg \text{MestoAktivno}(m,p)
	\end{aligned}
\end{equation}

\section{Definicije pojmov}
\begin{description}
	\item[MIR] MIR je kratica za \textit{Mid-level intermediate representation} in je bistveno poenostavljena oblika Rust-a.
	      Uporablja se za dele prevajalnika, ki zahtevajo informacijo o kontroli pretoka (toka?) \cite{MIRMidlevelIR}.
	      \subitem \textbf{Mesto (\textit{place})} je izraz, ki nam opredeli (identify?) lokacijo v pomnilniku \cite{MIRMidlevelIR}.
	      \citeauthor{weissOxideEssenceRust2019} primerjajo pojem mesta s pojmom \textit{lvalue} iz programskega jezika C.
	\item[Referenca] v Rust-u je podobna kazalcu v ostalih programskih jezikih, vendar prevajalnik zagotavlja, da kaže
	      na veljavno vrednost podanega tipa za celotno dobo aktivnosti reference (? pisalo je for the life of the reference) \cite{klabnikRustProgrammingLanguage2023}.
	      Reference ločimo na dva tipa \cite{crichtonGroundedConceptualModel2023,yanovskiGhostCellSeparatingPermissions2021,weissOxideEssenceRust2019}:
	      \subitem \textbf{Deljena referenca (\textit{shared / immutable reference})}: V programu lahko ustvarimo več deljenih referenc, ki
	      kažejo na isto mesto, vendar podatkov na tem mestu ne smemo spreminjati.
	      \subitem \textbf{Spremenljiva referenca (\textit{ unique / mutable reference})}: Da lahko mutiramo mesto na katerega kaže
	      referenca, ustvarimo spremenljivo referenco, ki je ena sama in nam omogoča spreminjanje vrednosti zapisane na posameznem mestu.
	      V literaturi jim večinoma pravijo \textit{unique references}, vendar smo izbrali prevod \textit{"spremenljiva"}, saj \textit{"edinstvena"} ne zveni ustrezno.
	      \subitem Tovrsten tip omejevanja ustvarjanja referenc se imenuje \textit{aliasing XOR mutability}. Ta model s pomočjo tipov
	      poveže podatke z dovoljenjimi operacijami, ki jih lahko izvajamo na teh podatkih \cite{yanovskiGhostCellSeparatingPermissions2021}.
	      \subitem \textbf{Deljena referenca (\textit{shared / immutable reference})}: V programu lahko ustvarimo več deljenih referenc
	      kažejo na isto mesto, vendar tega mesta potem ne smemo mutirati.
	      \subitem \textbf{Spremenljiva referenca (\textit{ unique / mutable reference})}: Da lahko mutiramo mesto na katerega kaže
	      referenca, pa ustvarimo spremenljivo referenco, ki je ena sama in nam omogoča spreminjanje vrednosti zapisane na posameznem mestu.
	      V literaturi jim večinoma pravijo \textit{unique}, vendar sem izbral prevod \textit{"spremenljiva"}, saj \textit{"edinstvena"} ne zveni ustrezno.
	      \subitem Tovrsten tip omejevanja ustvarjanja referenc se imenuje \textit{aliasing XOR mutability}. Ta model poveže podatke
	      z dovoljenjimi operacijami, ki jih lahko izvajamo nad temi podatki, s pomočjo tipov \cite{yanovskiGhostCellSeparatingPermissions2021}.
	\item[Življenska doba (\textit{lifetime})] Vsaka referenca ima življenjsko dobo, ki nam opiše doseg v programu znotraj katerega je
	      referenca še vedno veljavna (torej kaže na veljavno mesto) \cite{klabnikRustProgrammingLanguage2023}.
	\item[Posoja (\textit{loan})] je tesno povezana s sorodnim pojmom \textbf{izraz izposoje} (\textit{borrow expression}). Izraz izposoje
	      je jezikovni konstrukt, ki nam omogoča, da ustvarimo referenco (primer izraza izposoje bi bil\verb|&mut x|).
	      Rustov priročnik za prevajlnik \cite{MIRMidlevelIR} pojma \textit{borrow expression} ne definira, ampak ga uporabi tako:
	      \begin{displayquote}
		      An Rvalue is an expression that creates a value: in this case, the rvalue is a
		      mutable borrow expression, which looks like \verb|&mut <Place>|
	      \end{displayquote}
	      Rvalue pa je definiran z enumeratorjem \verb|Rvalue| \cite{RvalueRustc_middleMira}. Nas pa zanima specifično
	      varianta \verb|Rvalue::ref(Region<'tcx>, BorrowKind, Place<'tcx>)|, ki ustvari referenco tipa \verb|BorrowKind|
	      na mesto \verb|Place|.
	      Pojem \textit{borrow expression} pogosto uporabljajo \citeauthor{weissOxideEssenceRust2019} v svojem članku o formalizaciji
	      podmnožice Rust-a. Njihov način uporabe se sklada z našo definicijo.
	      \textbf{Posoja} (\textit{loan}) je interni konstrukt prevajalnika, ki hrani stanje o referenci in njenemu izvoru
	      \cite{weissOxideEssenceRust2019}. V trenutni implementaciji preverjalnika izposoj je izposoja predstavljena kot
	      urejena trojica \cite{2094nllRustRFC} \verb=('a, shared|uniq|mut, lvalue)=, kjer je:
	      \begin{itemize}
		      \item \verb|'a|: življenjska doba za katero je vrednost izposojena. To se nanaša na življenjske dobe kot
		            del Rustovega sistema tipov, ne pa kot množico izposoj, kot jih bomo definirali kasneje.
		      \item \verb=shared|uniq|mut=: tip izposoje
		      \item \verb|lvalue|: vrednost, ki je bila izposojena
	      \end{itemize}
	\item[NLL] je trenutna implementacija preverjevalnika izposoj v Rust-ovem prevajalniku.
\end{description}

\section{Osnovne množice in elementi}

\subsection{Množica posoj $\mathcal{L}$}
Množico vseh posoj (\textit{angl. loans}) označimo z $\mathcal{L}$. \textbf{Pogoji posoje} so lastnosti, ki morajo
držati v določeni točki programa, da smatramo posojo kot veljavno oz. aktivno.
Pravimo, da \textbf{razveljavimo pogoje posoje}, če velja ena izmed naslednjih točk:
\begin{itemize}
	\item Referenca je deljena (\textit{shared}), torej je oblike \verb|&x| in
	      \begin{itemize}
		      \item ustvarimo novo spremenljivo referenco
		      \item pišemo v mesto, ki je bilo izposojeno
	      \end{itemize}
	\item Referenca je spremenljiva in jo spreminjamo na kakršen koli način (ustvarjanje nove reference, pisanje, premikanje)
\end{itemize}

Ta pravila načeloma sledijo NLLm, bolj formalno jih opisujejo pravila razveljavljanja posoje (\textit{loan killed}).
Iz NLL RFC-ja \cite{2094nllRustRFC}:
\begin{displayquote}
	For a statement at point P in the graph, we define the "transfer function" – that is,
	which loans it brings into or out of scope – as follows:
	\begin{itemize}
		\item ...
		\item if this is an assignment \verb|lv = <rvalue>|, then any loan for some path P of which \verb|lv| is a prefix is killed.
	\end{itemize}
\end{displayquote}

kjer je pojem \textit{prefix} definiran tako:
\begin{displayquote}
	\textbf{Prefixes}. We say that the prefixes of an lvalue are all the lvalues you get by stripping away fields and derefs.
	The prefixes of \verb|*a.b| would be \verb|*a.b|, \verb|a.b|, and \verb|a|.
\end{displayquote}

V članku avtor pravi posojam tudi izrazi izposoje (\textit{borrow expressions}). V našem primeru bi se posoje ustvarile na naslednjih
mestih:

\begin{listing}[!ht]
	\begin{minted}{rust}
fn main() {
  let mut x: i32 = 22;
  let mut v: Vec<&'0 i32> = vec![];
  let r: &'1 mut Vec<&'2 i32> = &'3 mut v; // \fbox{posoja $L_{0}$}
  let p: &'5 i32 = &'4 x; // \fbox{posoja $L_{1}$}
  r.push(p);
  x += 1;
  take::<Vec<&'6 i32>>(v);
}
fn take<T>(p: T) { .. }
\end{minted}
	\caption{Posoje v programu}
	\label{listing:loans}
\end{listing}

\subsection{Množica regij $\mathcal{R}$}
V trenutni implementaciji preverjevalnika izposoj NLL se posoje spremljajo s pomočjo življenjskih dob.
V tej formulaciji pa je avtor življenjske dobe poimenoval regije (\textit{regions}).
Množica regij je označena z $\mathcal{R} \subset 2^{\mathcal{L}}$.
Na primeru so že označene z \verb|'1|, \verb|'2|, \verb|'3|, itd. Pripadnost posoj regijam bomo kasneje določili z relacijo.

\subsection{Graf poteka izposoje}
Graf poteka (\textit{angl. CFG - control flow graph}) je izračunan v prejšnjih fazah analize kode.
Zgrajen je iz osnovnih blokov, ti pa so zgrajeni iz stavkov. Spremlja ga tudi nekaj dodatnih informacij,
ki nam bodo kasneje prišle prav. Te so izračunane tekom analize poteka podatkov
(\textit{dataflow analysis})\cite{MIRDataflowRust}. Graf poteka označimo s $C = (C_{V}, C_{E})$, kjer
je $C_{V}$ množica vozlišč in $C_{E}$ množica povezav.

% TODO: Malo se preberi ali je res to pravilno. Kako dejansko dobimo input za CFG in

Privzeto se ustvarijo naslednje povezave (to bi moral potrditi v kodi rustc, v članku tako piše):

\begin{itemize}
	\item Za vsak stavek se ustvari povezava med njegovim začetkom in sredino:
	      $\forall stmt \in \mathcal{S}: (S(stmt), M(stmt)) \in C_{E}$
	\item Če $M(stmt)$ predstavlja \textit{terminator} (stavek na koncu bloka) potem dodamo povezavo iz njega v $S(stmt')$
	      za vsak stavek $stmt'$, ki mu sledi.
\end{itemize}

\textit{Opomba:} To je samo matematična formulacija predstavitve grafa poteka, v prevajalniku
je njegova predstavitev precej bolj kompleksna.

\subsubsection{Množica stavkov in točk}

Množico vseh stavkov v MIR označimo s $\mathcal{S}$. Točke v grafu poteka (\textit{CFG - control flow graph})
označimo s $\mathcal{P}$. Lahko so dveh tipov:
\begin{itemize}
	\item \textbf{na začetku stavka:} označuje trenutek preden se stavek izvede. Označimo s $S(stmt)$.
	\item \textbf{med stavkom:} označuje trenutek tik preden ima stavek učinek (v članku napisano "just before the statement takes effect").
	      Označimo s $M(stmt)$.
\end{itemize}

Avtor spletne objave ne opredeli pojmov \textit{na začetku stavka} in \textit{med stavkom} natančno,
vendar lahko najdemo razlago v \verb|legacy| implementaciji Polonius preverjevalnika izposoj.
Komentar nad strukturo, ki opisuje množico stavkov, pravi naslednje \cite{RustCompilerRustc_borrowcka}:

\begin{displayquote}
	Ta struktura prevede MIR lokacijo, ki identificira stavek znotraj osnovnega bloka, v "obogateno lokacijo",
	kar nam omogoči večjo granularnost. Bolj podrobno, ločimo med začetkom in sredino stavka. Sredina stavka
	je točka \textit{tik preden} ima stavek učinek. Torej za prirejanje \verb|A = B| bi bila sredina stavka
	točka trenutek ravno preden bi se \verb|B| zapisal v \verb|A| \dots
\end{displayquote}

\subsection{Začetne relacije}

Relacijo označimo kot \textbf{začetno} (\textit{input}), ko jo dobimo tako, da jo izpeljemo
iz direktno analize poteka podatkov. Te relacije bodo predstavljale našo izhodiščno točko iz katere bomo izpeljali druge relacije.


\subsubsection{Začetna relacija vsebovanosti}

Začetno relacijo vsebovanosti (\textit{base subset}) bomo označili z
$\beta \subset \mathcal{R} \times \mathcal{R} \times \mathcal{P}$. Torej to je relacija, ki povezuje dve regiji
ob neki točki v programu.

Bolj natančno, če velja $(R_{1}, R_{2}, P) \in \beta$ pomeni, da je $R_{1}$ podmnožica regije $R_{2}$ na točki $P$ v programu.
To dejstvo mora veljati na sredini stavka ($M(stmt)$), ki inducira zahtevo.

\textit{Opomba:} Oznaka \verb|<:| nam predstavlja vsebovanost med tipi (\textit{subtyping relation}).

\begin{lstlisting}
fn main() {
  let mut x: i32 = 22;
  let mut v: Vec<&'0 i32> = vec![];

  let r: &'1 mut Vec<&'2 i32> = &'3 mut v; 
  // tukaj zahtevamo naslednje: &'3 mut Vec<&'0 i32> <: &'1 mut Vec<&'2 i32> 
  (* $('3, '1, P) \in \beta, ('0, '2, P) \in \beta, ('2, '0, P) \in \beta$ *)

  let p: &'5 i32 = &'4 x; 
  // zahtevamo: &'4 i32 <: &'5 i32
  (* $('4, '5, P) \in \beta$ *)

  r.push(p);
  // zahtevamo: &'5 i32 <: &'2 i32
  (* $('5, '2, P) \in \beta$ *)

  x += 1;

  take::<Vec<&'6 i32>>(v);
  // zahtevamo Vec<&'0 i32> <: Vec<&'6 i32>
  (* $('0, '6, P) \in \beta$ *)
}
\end{lstlisting}

\subsubsection{Začetna relacija posoje regij}

Začetno relacijo posoje regij (\textit{borrow region}) označimo s $\psi \subseteq \mathcal{R} \times \mathcal{L} \times \mathcal{P}$.

Če velja $(R,L,P) \in \psi$ pomeni, da izraz izposoje na točki $P$ ustvari posojo $L$ in postane del
regije $R$. Prav tako kot relacija vsebovanosti se ta zahteva vzpostavi na sredini stavka.

\begin{lstlisting}
fn main() {
  let mut x: i32 = 22;
  let mut v: Vec<&'0 i32> = vec![];
  let r: &'1 mut Vec<&'2 i32> = &'3 mut v; 
  // $('3, L_{0}, P) \in \psi $
  let p: &'5 i32 = &'4 x; 
  // $('4, L_{1}, P) \in \psi $
  r.push(p);
  x += 1;
  take::<Vec<&'6 i32>>(v);
}
\end{lstlisting}

\subsubsection{Relacija aktivnosti regije}

Začetno relacijo aktivnosti regije (\textit{region live at}) označimo z $\nabla_{R} \subseteq \mathcal{R} \times \mathcal{P}$.
$(R, P) \in \nabla_{R}$ pomeni, da je regija $R$ aktivna na točki $P$. Torej spremenljivka, katere tip vključuje $R$ (recimo \verb|&'a|),
bo morda kasneje v programu uporabljena.

To določi analiza aktivnosti, ki poteka isto kot v NLL RFC. Bolj specifično, s pomočjo raznih omejitev izračuna množico točk
kjer mora biti regija (v RFC-ju poimenovana \textit{lifetime}) aktivna \cite{2094nllRustRFC}.

\subsubsection{Relacija prekinitve posoje}

Začetno relacijo prekinitve posoje (\textit{loan killed at}) označimo s $\kappa \subseteq \mathcal{L} \times \mathcal{P}$.
$(L,P) \in \kappa$ pomeni, da je posoja $L$ prekinjena (\textit{killed}) na točki $P$. Pojem prekinitve oziroma razveljavitve
pogojev smo definirali že zgoraj. To se običajno zgodi na sredini prireditvenega stavka, ki prepiše pot (\textit{path}) prej povezano s posojo $L$.

V našem primeru nimamo nobenega primera prekinitve posoje, je pa relacija ključna v naslednjem primeru:

\begin{listing}[!ht]
	\begin{minted}{rust}
let p = 22;
let q = 44;
let x: &mut i32 = &mut p; // `x` kaže na `p`
let y = &mut *x; // Posoja L0, `y` kaže tudi na `p`
// ...
x = &mut q; // `x` kaže na `q`; prekine L0
// Lahko uporabimo *x tukaj
\end{minted}
	\caption{Primer prekinitve posoje}
	\label{listing:loanKilled}
\end{listing}

V tem primeru je \verb|x| referenca na \verb|p|, ki je prekopirana v \verb|y|. Dostop do \verb|*x| bi bil tukaj neveljaven,
ker si ga je \verb|y| izposodil. Ko pa \verb|x| priredimo novo vrednost, pa razveljavimo posojo \verb|L0| in s tem si že spet
omogočimo dostop do \verb|x|. Brez prekinitve bi Polonius mislil, da je mesto \verb|*x| še vedno izposojeno, čeprav
zdaj \verb|y| kaže na \verb|p| in \verb|x| na \verb|q|.

\subsubsection{Relacija razveljavitve posoje}

Začetno relacijo razveljavitve posoje (\textit{invalidates loan}) označimo z $\iota \subset \mathcal{P} \times \mathcal{L}$.
To pomeni, da dejanje na točki $P$ (recimo mutacija izposojenega mesta) razveljavi pogoje posoje $L$, kar
je že opisano v poglavju o definicije množice $\mathcal{L}$.

\begin{listing}[!ht]
	\begin{minted}{rust}
fn main() {
  let mut x: i32 = 22;
  let mut v: Vec<&'0 i32> = vec![];
  let r: &'1 mut Vec<&'2 i32> = &'3 mut v; 
  // $('3, L_{0}, P) \in \psi $
  let p: &'5 i32 = &'4 x; 
  // $('4, L_{1}, P) \in \psi $
  r.push(p);
  x += 1; // tukaj razveljavimo $L_{1}$ z mutacijo deljenega referenta (ustvarili smo deljeno referenco v p)
  take::<Vec<&'6 i32>>(v);
}
\end{minted}
	\caption{Primer razveljavitve posoje}
	\label{listing:loanInvalidated}
\end{listing}

\subsection{Izpeljane relacije}

V tem poglavju bomo opisali relacije, ki jih izpeljemo iz začetnih. Te relacije tvorijo najbolj pomembni del analize
Polonius-a.

V primerih ne bomo označevali točk v grafu poteka pri relacijah, ker bo koda anotirana na tistem mestu, kjer
se posamezna relacija pojavi. V ozadju se to še vedno izvaja na nivoju MIR, vendar za naše poenostavljene
primere to ni ključna informacija. (torej pisali bomo $(R_{1}, R_{2}) \in \beta$ namesto $(R_{1}, R_{2}, P) \in \beta$).

\subsubsection{Relacija vsebovanosti}

Razširimo začetno relacijo vsebovanosti z (raširjeno) relacijo vsebovanosti (\textit{subset}), ki jo označimo z
$\Gamma \subseteq \mathcal{R} \times \mathcal{R} \times \mathcal{P}$. Definirana je z zaprtjem naslednjih pravil:

\begin{enumerate}
	\item \textbf{Začetna relacija: }Če $(R_1, R_2, P) \in \beta$, potem $(R_1, R_2, P) \in \Gamma$. Torej vse trojice
	      iz začetne relacije se pojavijo tudi v razširjeni.
	\item \textbf{Tranzitivnost:} Če $(R_1, R_2, P) \in \Gamma$ in $(R_2, R_3, P) \in \Gamma$, potem $(R_1, R_3, P) \in \Gamma$.
	      Relacija vsebovanosti na isti točki v programu je tranzitivna.
	\item \textbf{Propagacija:} Če veljajo vse izmed naštetega:
	      \begin{enumerate}
		      \item $(R_1, R_2, P) \in \Gamma$
		      \item $(P, Q) \in C_E$: točki si sledita v grafu poteka
		      \item $(R_1, Q) \in \nabla_R$: regija 1 je aktivna na naslednji točki
		      \item $(R_2, Q) \in \nabla_R$: regija 2 je aktivna
	      \end{enumerate}
	      potem sledi $(R_1, R_2, Q) \in \Gamma$. To pomeni, da se relacija propagira čez graf poteka, če sta obe
	      regiji aktivni na naslednji točki v grafu. Pogoj za aktivnost nam pride prav kasneje.
\end{enumerate}

Na primeru ustvarimo naslednje relacije:

\begin{listing}[!ht]
	\begin{minted}{rust}
fn main() {
  let mut x: i32 = 22;
  let mut v: Vec<&'0 i32> = vec![];

  let r: &'1 mut Vec<&'2 i32> = &'3 mut v;
  // $('3, '1) \in \beta, ('0, '2) \in \beta, ('2, '0) \in \beta$

  let p: &'5 i32 = &'4 x;
  // $('3, '1) \in \beta, ('0, '2) \in \beta, ('2, '0) \in \beta$
  // $('4, '5) \in \beta$

  r.push(p);
  // $('3, '1) \in \beta, ('0, '2) \in \beta, ('2, '0) \in \beta$
  // $('4, '5) \in \beta$
  // $('5, '2) \in \beta$
  
  x += 1;

  take::<Vec<&'6 i32>>(v);
  // $('3, '1) \in \beta, ('0, '2) \in \beta, ('2, '0) \in \beta$
  // $('4, '5) \in \beta$
  // $('5, '2) \in \beta$
  // $('0, '6) \in \beta$
}

fn take<T>(p: T) { .. }
\end{minted}
	\caption{Relacija vsebovanosti}
	\label{listing:subsetRelations}
\end{listing}

\subsubsection{Relacija zahteve}

Relacija zahteve nam pove, da regija $R$ zahteva, da pogoji posoje $L$ veljajo na točki $P$. Označimo jo s
$\zeta \subseteq \mathcal{R} \times \mathcal{L} \times \mathcal{P}$ in definirana je z zaprtjem naslednjih pravil:
\begin{enumerate}
	\item \textbf{Začetna relacija:} Če $(R, L, P) \in \psi$, potem $(R, L, P) \in \zeta$. To nam pove, da če se trojica
	      nahaja v relaciji posoje regij, se nahaja tudi v $\zeta$.
	\item \textbf{Vsebovanost:} Če velja $(R_{1}, L, P) \in \zeta$ in $(R_{1}, R_{2}, P) \in \Gamma$, potem sledi
	      $(R_{2}, L, P) \in \zeta$. To nam pove, da če neka regija $R_{1}$, ki je podmnožica večje regije $R_{2}$, na točki $P$
	      zahteva posojo $L$, potem tudi $R_{2}$ zahteva isto posojo.
	\item \textbf{Propagacija:} Če veljajo vse \begin{enumerate}
		      \item $(R,L,P) \in \zeta$: $R$ zahteva $L$ na $P$
		      \item $(L, P) \notin \kappa$: $L$ ni prekinjena na $P$
		      \item $(P, Q) \in C_{E}$: $Q$ sledi $P$ v grafu poteka
		      \item $(R,Q) \in \nabla_{R}$: regija $R$ je aktivna na točki $Q$
	      \end{enumerate}
	      potem propagiramo relacijo v $(R,L,Q) \in \zeta$.
\end{enumerate}

Opazimo, da pri relaciji vsebovanosti $\beta$ in pri relaciji zahteve $\zeta$ mora biti regija pri pravilu
za propagacijo aktivna na naslednji točki $Q$. Z naslednjim primerom ponazorimo zakaj je to ključna omejitev.

\begin{listing}[!ht]
	\begin{minted}{rust}
let x = 22;
let y = 44;

let mut p: &'0 i32 = &'1 x; // posoja $L_{0}$
// $('1,'0) \in \beta$
// $('1, L_{0}) \in \zeta$

p = &'3 y; // posoja $L_{1}$
// $('3,'0) \in \beta$
// $('3, L_{1}) \in \zeta$
// '1 ni več aktivna, ker smo jo prepisali z '3 

x += 1;
// razveljavi se posoja $L_{0}$ $(L_{0}) \in \iota$
// tukaj bi brez pravila o aktivnosti regij še vedno zahtevali 
// $(L_{0}, '0) \in \zeta$ zaradi pravila o propagaciji 

print( *p );
// ta izraz posledično ne bi bil veljaven
\end{minted}
	\caption{Primer relacije zahteve}
	\label{listing:reqRelation}
\end{listing}

\subsubsection{Relacija aktivnosti posoje}

Relacija aktivnosti posoje (\textit{loan live at}) pomeni, da je posoja $L$ aktivna na točki $P$. Označimo jo s
$\nabla_{L} \subseteq \mathcal{L} \times \mathcal{P}$ in jo definiramo takrat, ko

\[ \exists R \in \mathcal{R}: (R,P) \in \nabla_{R} \land (R,L,P) \in \zeta \]

To na kratko pomeni, da je posoja aktivna, če jo na isti točki zahteva neka aktivna regija.

\subsection{Javljanje napake}

S pomočjo prejšnjih relacij lahko na koncu definiramo kje v programu javimo napako (v obsegu preverjalnika posoj).
Že spet si pomagamo z relacijo, ki jo tokrat poimenujemo \textbf{relacija napake} (\textit{error}) in jo označimo z
$\epsilon \subset \mathcal{P}$. Ta relacija nam pove, da javimo napako na točki $P$ v programu.

Definiramo jo, ko velja:

\[ \exists L \in \mathcal{L}: (P, L) \in \iota \land (L,P) \in \nabla_{L} \]

Torej napaka se javi natanko tedaj, ko neko dejanje na točki $P$ razveljavi pogoje posoje $L$, ki je hkrati tudi
aktivna na točki $P$.

Poglejmo še kako se dokončno napaka javi na našem primeru:

\begin{listing}[!ht]
	\begin{minted}{rust}
fn main() {
  let mut x: i32 = 22;
  let mut v: Vec<&'0 i32> = vec![];

  let r: &'1 mut Vec<&'2 i32> = &'3 mut v;
  // relacije, ki so ustvarjene tukaj niso relevantne za napako 

  let p: &'5 i32 = &'4 x;
  // $('4, L_{1}) \in \zeta $
  // $('4, '5) \in \beta \implies ('5, L_{1}) \in \zeta$

  r.push(p);
